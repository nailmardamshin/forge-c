Зачем это делали

Документооборот должен быть контролирующей функцией, а не ручной работой

Отдел — 18 человек, ФОТ ~1.15М ₽/мес, в сезон +2-3. Масштаб только через найм. Пять болей с брифинга:

  • Ручной труд на всех этапах
  • Нет единого расчёта → ошибки в счетах, налоговые риски
  • Данные разбросаны: 1С / Amo / таблицы / переписки
  • Нет аналитики — отвалы считают неделю руками
  • Сложность управления коллективом
Цель проекта: 18 → 3-5 человек, рост объёмов без расширения штата
Что построили

Шаг 1 — оформление договоров текущих клиентов

90% потока отдела в сезон (~1620 задач/мес). Полный цикл автоматически:

заявка (Яндекс.Трекер / Google-таблица) → расчёт цены и скидок → договор + приложение + счёт → заказ в 1С + задача оператору
  • Человек в контуре (supervised): автоматика готовит всё, сотрудник проверяет и жмёт «Отправить». Не замена одного ручного труда другим.
  • Под капотом: интеграции 1С / Amo / Трекер / Google, свой сервер, мониторинг, алерты, журнал каждой операции.

Сложности и чему научились

Запускались на живом потоке — и это правильно

  • Два канала заявок (форма + таблица) обрабатываются по-разному — выявили и выравниваем на реальных данных.
  • Нюансы документов видны только в бою, не на тестах — ловим по обратной связи операторов.
  • Корректировки под согласованные шаблоны и изменения: правила расчёта уточнялись в процессе (в т.ч. при смене Якимова → Виолетта) — вносим на ходу, без остановки потока.
  • Supervised-режим оправдался — человек в контуре ловит нетиповое, нулевой риск «тихих» ошибок клиенту.
Система зрелая — не «сломалось», а «дообкатываем под реальный поток»
